## **1\. 개요 (Executive Summary)**

본 프로젝트는 Next.js 백엔드와 React 프론트엔드, MySQL을 활용한 영화 추천 플랫폼입니다. Docker를 활용한 개발 환경 구성, 윌슨 점수(Wilson Score)를 도입한 추천 알고리즘 등 대학 학부 수준을 상회하는 기술적 완성도를 갖추고 있습니다.

그러나 실제 서비스 배포 및 운영 환경을 가정했을 때 **보안(Security)**, **아키텍처(Architecture)**, **사용자 경험(UX)** 측면에서 보완이 필요한 사항들이 식별되었습니다. 본 제안서는 이를 개선하여 프로젝트의 기술적 가치를 높이는 것을 목표로 합니다.

## **2\. 주요 개선 사항 (Key Improvements)**

### **2.1. 보안 및 인증 강화 (Security)**

가장 시급하게 해결해야 할 기술적 부채로, 실제 서비스 시 심각한 위협이 될 수 있는 항목입니다.

#### **① 비밀번호 해싱 알고리즘 변경**

* **현황:** backend/lib/password.ts에서 SHA-256 단방향 해시를 사용 중.  
* **개선안:** **bcrypt** 또는 **Argon2** 라이브러리로 교체하고 Salt를 적용.  
* **기술적 근거:** SHA-256은 연산 속도가 매우 빨라, GPU를 이용한 무차별 대입 공격(Brute-force Attack)이나 레인보우 테이블 공격에 취약합니다. 비밀번호 저장에는 의도적으로 연산 속도를 늦추는 알고리즘(Key Stretching)이 필수적입니다.

#### **② 관리자 권한 관리 방식 변경**

* **현황:** 프론트엔드 환경 변수(VITE\_ADMIN\_IMPORT\_TOKEN)에 관리자 토큰이 포함되어 클라이언트 빌드 시 노출됨.  
* **개선안:** 클라이언트의 토큰 검증 로직을 제거하고, **DB의 Users 테이블에 role 컬럼(admin/user)을 추가**하여 백엔드 세션/JWT에서 권한을 확인하도록 변경.  
* **기술적 근거:** 프론트엔드 코드는 브라우저에서 누구나 열람 가능하므로, 민감한 관리자 토큰을 두어서는 안 됩니다. 권한 제어는 반드시 신뢰할 수 있는 서버(Backend)에서 수행되어야 합니다.

#### **③ 인증 상태 유지 (Persistence)**

* **현황:** React State(user)에만 로그인 정보를 저장하여, 새로고침 시 로그인이 풀림.  
* **개선안:** **JWT(Access/Refresh Token)** 방식을 도입하거나, 보안 쿠키(HttpOnly Cookie)를 활용한 세션 관리 도입.  
* **기술적 근거:** 사용자 경험(UX) 측면에서 로그인이 유지되어야 하며, 보안 측면에서도 클라이언트 조작이 불가능한 HttpOnly 쿠키 등을 사용하는 것이 XSS 공격 방어에 유리합니다.

### **2.2. 프론트엔드 아키텍처 리팩토링 (Architecture)**

기능 구현 위주로 작성된 코드를 유지보수가 용이한 구조로 변경합니다.

#### **① 관심사 분리 (Separation of Concerns)**

* **현황:** App.tsx 파일 하나에 라우팅, API 호출, 상태 관리, UI 렌더링이 모두 포함된 'God Component' 형태.  
* **개선안:**  
  * **Routing:** react-router-dom을 도입하여 페이지 이동을 URL 기반으로 관리 (뒤로 가기 기능 활성화).  
  * **Hooks:** useMovies, useAuth 등 커스텀 훅으로 비즈니스 로직 분리.  
  * **Context:** 전역 상태(User, Theme 등)를 Context API 또는 Zustand로 분리.  
* **기술적 근거:** 단일 파일의 코드 라인 수를 줄여 가독성을 높이고, 로직과 UI를 분리하여 테스트 용이성 및 재사용성을 확보해야 합니다.

#### **② 검색 로직의 서버 사이드 이관**

* **현황:** 전체 영화 데이터를 프론트엔드로 가져온 후 filter() 함수로 검색 수행.  
* **개선안:** 백엔드 API에 검색 파라미터(GET /api/movies?q=keyword)를 추가하고, **DB 레벨에서 WHERE LIKE 절이나 Full-text Search**를 수행하여 결과만 반환.  
* **기술적 근거:** 데이터가 수천 건 이상으로 늘어날 경우, 클라이언트의 메모리 부하가 커지고 초기 로딩 속도가 급격히 느려집니다. 대용량 처리를 위해 필터링은 DB에서 수행해야 합니다.

---

### **2.3. 백엔드 및 데이터베이스 (Backend & DB)**

안정성과 데이터 무결성을 위한 개선입니다.

#### **① ORM 도입 (생산성 및 안정성)**

* **현황:** mysql2 라이브러리를 사용하여 Raw SQL 쿼리를 직접 문자열로 작성.  
* **개선안:** **Prisma** 또는 **Drizzle ORM** 도입.  
* **기술적 근거:** 문자열 기반 쿼리는 오타로 인한 런타임 에러 위험이 높습니다. ORM을 사용하면 타입 안정성(Type-safety)을 확보할 수 있고, 복잡한 JOIN 쿼리나 마이그레이션 관리가 수월해집니다.

#### **② 트랜잭션 범위 확대**

* **현황:** 데이터 임포트 시에는 트랜잭션이 적용되어 있으나, 리뷰 작성 및 평점 집계 갱신 로직의 원자성(Atomicity) 보장이 불명확함.  
* **개선안:** 리뷰 INSERT와 영화 평점 통계 UPDATE 로직을 **단일 트랜잭션**(conn.beginTransaction)으로 명확히 묶어 처리.  
* **기술적 근거:** 리뷰는 저장되었으나 통계 갱신에 실패할 경우, 데이터 불일치(Inconsistency)가 발생할 수 있습니다.

### **2.4. 차별화 기능 완성 (UX & Features)**

프로젝트의 포트폴리오 가치를 높이는 '한 방'입니다.

#### **① AI Chat API 실연동**

* **현황:** 현재 mock 데이터로 더미 응답만 반환 중.  
* **개선안:** **OpenAI API** 또는 **Google Gemini API**를 연동.  
  * System Prompt에 현재 DB 스키마나 영화 목록 요약을 주입하여 "놀란 감독 영화 추천해줘" 같은 질문에 실제 데이터 기반 답변을 하도록 구현 (RAG 기술의 기초).  
* **기술적 근거:** 단순 CRUD를 넘어 최신 트렌드인 AI 기술을 서비스에 접목했다는 강력한 어필 포인트가 됩니다.
